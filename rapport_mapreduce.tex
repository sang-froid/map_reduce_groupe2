\section{Introduction et contexte du problème}

Dans le domaine de l'Internet des Objets (IoT), des milliers de capteurs répartis géographiquement mesurent en continu des grandeurs physiques : température, humidité, vibration et trafic réseau. L'enjeu est de \textbf{regrouper automatiquement} ces capteurs en clusters homogènes selon leurs profils de mesure, afin d'identifier des zones géographiques aux comportements similaires.

\medskip
Ce TP4 implémente l'algorithme \textbf{K-Means} dans trois variantes progressives :

\begin{enumerate}[leftmargin=2cm]
  \item \textbf{K-Means classique} (séquentiel, mono-nœud)
  \item \textbf{K-Means MapReduce} (parallèle, multiprocessing Python)
  \item \textbf{K-Means Streaming} (traitement continu de vagues de données)
\end{enumerate}

\medskip
Le \textbf{scénario IoT} retenu est le suivant : cinq serveurs régionaux (Nord, Centre, Sud, Ouest, Est) stockent chacun les mesures de 300 capteurs locaux, soit 1 500 capteurs au total. Pour les tests de charge, un dataset de 50 000 capteurs est également généré.

\subsection{Modèle de parallélisation}

L'exécution séquentielle du K-Means devient un goulot d'étranglement dès que le volume de données augmente : chaque itération doit parcourir \emph{tous} les points sur \emph{un seul processeur}. La modélisation parallèle repose sur l'observation suivante :

\begin{mdframed}[linecolor=black!30,linewidth=0.8pt]
\textbf{Principe clé :} L'étape d'affectation de chaque capteur à son cluster le plus proche est \emph{totalement indépendante} d'un capteur à l'autre. On peut donc distribuer les points sur $N$ workers, chacun traitant un sous-ensemble (chunk) de manière parallèle.
\end{mdframed}

\medskip
Deux modes de partitionnement ont été implémentés :

\begin{description}[leftmargin=2cm]
  \item[\textbf{Mode chunk}] Un seul fichier CSV global est découpé en $N$ blocs de taille égale. Chaque worker traite son bloc indépendamment.
  \item[\textbf{Mode multi-serveurs}] Chaque fichier régional (nord, centre, sud, ouest, est) est assigné à un worker distinct — simulation fidèle du scénario IoT où les données restent locales au serveur.
\end{description}

\section{Explication des Mappers, Combiners, Shuffle et Reducers}

\subsection{Mapper + Combiner (\texttt{mapper\_combiner})}

\begin{mdframed}[linecolor=black!30,linewidth=0.8pt]
\textbf{Entrée :} un chunk de lignes CSV brutes + les $K$ centroïdes courants\\
\textbf{Sortie :} \texttt{dict\{ cluster\_id : (somme\_partielle, count) \}}
\end{mdframed}

\medskip
Pour chaque capteur du chunk, le mapper :
\begin{enumerate}
  \item Parse la ligne CSV et extrait le vecteur \texttt{[temp, hum, vib, net]}.
  \item Applique éventuellement la normalisation z-score en ligne.
  \item Calcule la \textbf{distance euclidienne} aux $K$ centroïdes courants.
  \item Assigne le capteur au cluster du centroïde le plus proche (\emph{étape MAP}).
  \item Accumule la somme partielle et le compteur local pour ce cluster (\emph{étape COMBINE}).
\end{enumerate}

Le \textbf{Combiner} est une optimisation critique : plutôt que d'émettre un enregistrement par capteur (potentiellement des millions), le worker n'émet qu'une somme partielle par cluster. Cela réduit considérablement le volume de données échangées lors du Shuffle.

\medskip
\textbf{Formule d'affectation :}
\[
  \text{cluster}(p) = \arg\min_{k} \; \| p - c_k \|_2
  \qquad \text{où } \|a - b\|_2 = \sqrt{\sum_{d} (a_d - b_d)^2}
\]

\subsection{Shuffle (\texttt{shuffle})}

\begin{mdframed}[linecolor=black!30,linewidth=0.8pt]
\textbf{Entrée :} liste des résultats partiels de tous les mappers\\
\textbf{Sortie :} \texttt{dict\{ cluster\_id : [(somme\_partielle, count), ...] \}}
\end{mdframed}

La phase de Shuffle regroupe toutes les sommes partielles émises par les différents workers par clé \texttt{cluster\_id}. Elle est simulée en mémoire (pas de réseau réel), ce qui est cohérent avec une exécution sur un seul nœud multi-cœur.

\subsection{Reducer (\texttt{reducer})}

\begin{mdframed}[linecolor=black!30,linewidth=0.8pt]
\textbf{Entrée :} \texttt{(cluster\_id, [(somme\_partielle, count), ...])}\\
\textbf{Sortie :} \texttt{(cluster\_id, nouveau\_centroïde, count\_total)}
\end{mdframed}

Pour chaque cluster, le reducer agrège les sommes partielles de tous les workers :

\[
  c_k^{\text{nouveau}} = \frac{\sum_{w} \text{somme}_w^{(k)}}{\sum_{w} \text{count}_w^{(k)}}
\]

où $w$ indexe les workers (serveurs régionaux). Ce calcul est exact : la somme des sommes partielles divisée par le count total est strictement équivalente à la moyenne globale.

\subsection{Initialisation K-Means++}

Les centroïdes initiaux ne sont pas choisis aléatoirement mais avec l'heuristique \textbf{K-Means++} : chaque nouveau centroïde est choisi avec une probabilité proportionnelle au carré de sa distance au centroïde le plus proche déjà sélectionné. Cela évite les mauvaises convergences vers des minima locaux sous-optimaux.

\subsection{Cœur MapReduce : \texttt{kmeans\_mr\_job.py}}

\begin{lstlisting}[caption={Mapper + Combiner (kmeans\_mr\_job.py)},label={lst:mapper}]
def mapper_combiner(args):
    chunk_lines, centroids = args[0], args[1]
    norm_params = args[2] if len(args) == 3 else None
    partial = {}

    for line in chunk_lines:
        line = line.strip()
        if not line or line.startswith("sensor_id"):
            continue
        parts = line.split(",")
        point = [float(parts[i]) for i in FEATURE_INDICES]  # temp,hum,vib,net

        # Normalisation z-score inline (optionnelle)
        if norm_params is not None:
            means_n, stds_n = norm_params
            point = [(point[d]-means_n[d])/stds_n[d] for d in range(len(point))]

        # MAP : cluster le plus proche
        distances = [euclidean(point, c) for c in centroids]
        cluster_id = distances.index(min(distances))

        # COMBINE : accumulation locale
        if cluster_id not in partial:
            partial[cluster_id] = ([0.0]*len(point), 0)
        s, cnt = partial[cluster_id]
        partial[cluster_id] = ([s[d]+point[d] for d in range(len(point))], cnt+1)

    return partial   # { cluster_id: (somme_partielle, count) }
\end{lstlisting}

\begin{lstlisting}[caption={Reducer (kmeans\_mr\_job.py)},label={lst:reducer}]
def reducer(cluster_id, values):
    # values = [(somme_partielle, count), ...] de chaque mapper
    total_sum, total_count = None, 0
    for partial_sum, count in values:
        if total_sum is None:
            total_sum = list(partial_sum)
        else:
            total_sum = [a + b for a, b in zip(total_sum, partial_sum)]
        total_count += count

    new_centroid = [s / total_count for s in total_sum]
    return cluster_id, new_centroid, total_count
\end{lstlisting}

\subsection{Orchestration d'une itération : \texttt{run\_mapreduce\_iteration}}

\begin{lstlisting}[caption={Orchestrateur MapReduce (kmeans\_mr\_job.py)},label={lst:orchestrator}]
def run_mapreduce_iteration(lines, centroids, n_workers=None, norm_params=None):
    # Decoupage en chunks (mode fichier) ou passage direct (mode multi-serveurs)
    if lines and isinstance(lines[0], list):
        chunks = lines              # mode multi-serveurs : 1 liste par serveur
    else:
        chunk_size = max(1, len(lines) // n_workers)
        chunks = [lines[i:i+chunk_size] for i in range(0, len(lines), chunk_size)]

    args = [(chunk, centroids, norm_params) for chunk in chunks] \
           if norm_params else [(chunk, centroids) for chunk in chunks]

    # MAP + COMBINE : parallele via multiprocessing.Pool
    with multiprocessing.Pool(processes=n_workers) as pool:
        partial_results = pool.map(mapper_combiner, args)

    # SHUFFLE
    shuffled = shuffle(partial_results, len(centroids))

    # REDUCE
    new_centroids = [None] * len(centroids)
    cluster_counts = {}
    for cluster_id, values in shuffled.items():
        if values:
            cid, centroid, count = reducer(cluster_id, values)
            new_centroids[cid] = centroid
            cluster_counts[cid] = count
    return new_centroids, cluster_counts
\end{lstlisting}

\medskip
\textbf{Points clés du code :}
\begin{itemize}
  \item \textbf{Ligne 10 :} \texttt{multiprocessing.Pool} lance réellement des processus OS en parallèle, exploitant plusieurs cœurs CPU simultanément.
  \item \textbf{Convergence :} Après chaque itération, on vérifie que $\|c_k^{\text{ancien}} - c_k^{\text{nouveau}}\|_2 < \epsilon$ pour tout $k$ (tolérance $\epsilon = 10^{-4}$).
  \item \textbf{Cluster vide :} Si un cluster perd tous ses points, l'ancien centroïde est conservé (ligne de sécurité dans l'orchestrateur).
\end{itemize}

\section{Exemple d'exécution sur un cas simple}

\subsection{Données d'entrée}

Considérons 6 capteurs répartis sur 2 serveurs (3 par serveur), à clusteriser en $K=2$ groupes :

\begin{center}
\begin{tabular}{lcccc}
\toprule
\textbf{Capteur} & \textbf{Temp (°C)} & \textbf{Hum (\%)} & \textbf{Vib (m/s²)} & \textbf{Net (kbps)} \\
\midrule
S1 (Serveur A) & 35.0 & 80.0 & 0.5 & 120  \\
S2 (Serveur A) & 34.5 & 82.0 & 0.4 & 110  \\
S3 (Serveur A) & 22.0 & 45.0 & 2.5 & 500  \\
S4 (Serveur B) & 36.0 & 79.0 & 0.6 & 130  \\
S5 (Serveur B) & 21.0 & 43.0 & 2.8 & 480  \\
S6 (Serveur B) & 23.0 & 47.0 & 2.3 & 510  \\
\bottomrule
\end{tabular}
\end{center}

Les 3 premiers capteurs représentent des zones chaudes et humides (profil forêt), les 3 derniers représentent des zones urbaines actives.

\subsection{Itération 1 — Phase MAP + COMBINE (Serveur A)}

Centroïdes initiaux (K-Means++) : $C_0 = [35.0, 80.0, 0.5, 120]$, $C_1 = [22.0, 45.0, 2.5, 500]$

\medskip
Le \textbf{Mapper A} traite S1, S2, S3 :

\begin{center}
\small
\begin{tabular}{lcc}
\toprule
Capteur & $d(p, C_0)$ & $d(p, C_1)$ \\ \midrule
S1 & \textbf{0.78} & 388.6 $\Rightarrow$ cluster 0 \\
S2 & \textbf{2.91} & 390.2 $\Rightarrow$ cluster 0 \\
S3 & 388.6 & \textbf{0.00} $\Rightarrow$ cluster 1 \\
\bottomrule
\end{tabular}
\end{center}

Après COMBINE local (Serveur A) :
\[
\text{partial}_A = \{ 0 : ([69.5, 162, 0.9, 230], \; 2), \quad 1 : ([22.0, 45.0, 2.5, 500], \; 1) \}
\]

\subsection{Itération 1 — Phase MAP + COMBINE (Serveur B)}

Le \textbf{Mapper B} traite S4, S5, S6 de manière \textbf{simultanée} (en parallèle) :

\begin{center}
\small
\begin{tabular}{lcc}
\toprule
Capteur & $d(p, C_0)$ & $d(p, C_1)$ \\ \midrule
S4 & \textbf{1.41} & 378.8 $\Rightarrow$ cluster 0 \\
S5 & 379.5 & \textbf{21.0} $\Rightarrow$ cluster 1 \\
S6 & 388.5 & \textbf{11.2} $\Rightarrow$ cluster 1 \\
\bottomrule
\end{tabular}
\end{center}

Après COMBINE local (Serveur B) :
\[
\text{partial}_B = \{ 0 : ([36.0, 79.0, 0.6, 130], \; 1), \quad 1 : ([44.0, 90.0, 5.1, 990], \; 2) \}
\]

\subsection{Phase SHUFFLE}

Le Shuffle regroupe les partiels par \texttt{cluster\_id} :
\[
\text{shuffled} = \{ 0 : [\text{partial}_A[0], \text{partial}_B[0]], \quad 1 : [\text{partial}_A[1], \text{partial}_B[1]] \}
\]

\subsection{Phase REDUCE}

Le Reducer calcule les nouveaux centroïdes :

\[
C_0^{\text{new}} = \frac{[69.5, 162, 0.9, 230] + [36.0, 79.0, 0.6, 130]}{2 + 1} = [35.17, 80.33, 0.50, 120]
\]
\[
C_1^{\text{new}} = \frac{[22.0, 45.0, 2.5, 500] + [44.0, 90.0, 5.1, 990]}{1 + 2} = [22.00, 45.00, 2.53, 496.7]
\]

\subsection{Résultat final}

Après convergence (2 itérations sur cet exemple) :

\begin{center}
\begin{tabular}{clp{7cm}}
\toprule
\textbf{Cluster} & \textbf{Centroïde final} & \textbf{Interprétation} \\
\midrule
$C_0$ & [35.2, 80.3, 0.50, 120] & \textbf{Zone chaude et humide} (forêt/rural) : température élevée, faible vibration, faible trafic réseau \\[4pt]
$C_1$ & [22.0, 45.0, 2.53, 496] & \textbf{Zone urbaine active} : température modérée, forte vibration (machines), trafic réseau élevé \\
\bottomrule
\end{tabular}
\end{center}

\textbf{Capteurs S1, S2, S4} → Cluster 0 (zones forestières/rurales chaudes)\\
\textbf{Capteurs S3, S5, S6} → Cluster 1 (zones urbaines industrielles)

\medskip
Le résultat est \textbf{identique} à ce qu'aurait produit l'algorithme classique séquentiel, mais en utilisant 2 processus parallèles simultanés.

\subsubsection{Petit dataset (1 500 capteurs) : overhead MapReduce}

Sur 1 500 points, le K-Means classique est \textbf{1.6× plus rapide} (0.126 s vs 0.203 s). Ceci s'explique par le \textbf{coût fixe de démarrage} de \texttt{multiprocessing.Pool} : créer des processus OS, sérialiser/désérialiser les arguments et résultats (pickle) représente une surcharge non négligeable face à un volume de données faible.

\subsubsection{Grand dataset (50 000 capteurs) : avantage MapReduce}

Sur 50 000 capteurs, MapReduce est \textbf{1.97× plus rapide} (1.293 s vs 2.548 s). Le traitement parallèle sur plusieurs cœurs amortit largement le coût de démarrage.

\section{Conclusion}

Ce TP a démontré l'intérêt concret du paradigme MapReduce pour le clustering de données IoT distribuées. Les principaux enseignements sont :

\begin{enumerate}
  \item \textbf{Équivalence mathématique} : L'algorithme MapReduce produit exactement les mêmes centroïdes que l'algorithme classique. La décomposition en sommes partielles est une transformation exacte, pas une approximation.

  \item \textbf{Seuil de rentabilité} : L'overhead de démarrage des processus parallèles rend MapReduce moins performant sur les petits volumes ($< 20\,000$ capteurs). Au-delà, le gain en parallélisme devient dominant.

  \item \textbf{Adéquation au scénario IoT} : Dans un contexte réel, les données sont déjà physiquement stockées sur les serveurs régionaux. Le paradigme MapReduce exploite cette localité : seules les \emph{sommes partielles} (vecteurs de 4 flottants) transitent sur le réseau, et non les données brutes.

  \item \textbf{Passage à l'échelle} : Au-delà de 500 000 capteurs, l'algorithme classique ne peut plus terminer dans un temps raisonnable, tandis que MapReduce continue à converger. C'est le cas d'usage pour lequel le paradigme a été conçu.
\end{enumerate}

\medskip
\begin{mdframed}[linecolor=black!30,linewidth=0.8pt]
\textbf{En production IoT réelle}, MapReduce apporterait des gains encore plus importants, car les calculs locaux sur chaque serveur régional éliminent complètement le transfert des données brutes. Seul un vecteur de 4 valeurs par cluster est envoyé au reducer, quelle que soit la taille du dataset local.
\end{mdframed}
